**33 pending entries total.** I traced what happened immediately after

| Pattern | Count | Description |
|---|---|---|
| Stage advance without close | ~20 | impl-review → coding etc. Work completed (next stage ran and produced artifacts) but log-response never written for the previous stage |
| Local work (`to: claude`) | ~12 | Orchestrator did impl-review/incorporate/ad-hoc itself. Hard Rule 5 says log it; orchestrator isn't closing those entries |
| Same-pane rapid re-dispatch | ~8 | Multiple dispatches to same pane within seconds/minutes. Likely `dispatch` where `send --force` was the right tool |
| Genuine silent stalls | **0 clear examples** | Every pending entry has a subsequent entry within a reasonable window; no evidence of "worker hung, pipeline froze" in this data |

(Categories overlap — `to: claude` entries are also stage-advance cases.)

### Key code reference

`cmd_callback` already auto-calls `cmd_log_response`
(`~/bin/forge-bridge:1484-1485`). When a worker calls
`forge-bridge callback --status DONE`, the log entry should close
automatically. So when do we get orphans?

1. Worker didn't call `callback` (template doesn't include it, or
   dispatch went via raw `send`)
2. `to: claude` local work — no worker to call callback; orchestrator
   must do it manually
3. Stage transitions — orchestrator advanced before writing log-response

The auto-arm proposal targets failure mode #1 (and only the variant
where the worker also stalls). Per the data, that's not the dominant
cause of pending entries — orchestrator log-response discipline is.

### Proposed fixes (my hand-rolled draft, NOT vetted)

1. **Tighten the log-before-send hook** to apply to `dispatch` too:
   refuse when the target has a prior pending entry. Add
   `dispatch --supersede` for legitimate redirects. (~45 min) — would
   have prevented rapid re-dispatch pattern.
2. **Require log-response before stage advance.** New Hard Rule + a
   hook in `dispatch` that refuses to advance to the next stage if the
   previous stage's entry (same slug, any worker) is still pending.
   (~30 min) — would have prevented `to: claude` orphans.
3. **Only build the auto-arm watchdog AFTER 1+2 ship** and only if
   evidence of true silent stalls emerges. Based on this data,
   probably not needed.
